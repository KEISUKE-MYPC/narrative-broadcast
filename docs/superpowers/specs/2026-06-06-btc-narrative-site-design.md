# 設計仕様：BTCナラティブ分析 配信サイト（Next.js + Vercel）

作成日：2026-06-06
ステータス：設計確定（実装計画 待ち）

## 1. 目的

GitHubに自動蓄積される6hナラティブ分析記事を、**独自の配信サイト**として公開する。
ルーチンの自動公開パイプラインにそのまま乗せ、**生成→公開→ライブ反映を完全自動**にする。

## 2. 背景・前提（確定事項）

- 読者層：日本語リテール中心 ／ 収益化：まず無料で読者基盤 ／ 運用：最小負荷・自動寄り
- コンテンツは既に `articles/` にMarkdownで蓄積。ルーチンが6時間ごと（JST 3/9/15/21時）に main へ自動公開。
- 配信スタイル：**6h記事をそのまま流すストリーム**＋将来の週次まとめ。
- 本拠地はこのサイト。X＝集客導線、note＝任意のクロス投稿（別途）。
- ドメイン：独自ドメインは後付け（まずVercel仮URLで開始）。

## 3. アーキテクチャ（採用案：同一リポジトリ＋SSG）

Next.jsアプリを `narrative-broadcast` リポジトリに同居させ、ビルド時に `articles/` のMarkdownを読む。

```
ルーチンが記事を main へ push → Vercel が自動再デプロイ → ライブサイトに反映
```

- DB不要・サーバ管理不要（Vercel Hobby／サーバーレス／自動SSL・CDN）。
- 1日4ビルド（6h×4）はHobby枠で十分。
- ルーチンは `articles/` のみ更新するため、アプリコードと競合しない。

**却下案**：別リポジトリ＋GitHub API取得（webhook/トークンで複雑化）、CMS/contentlayer（過剰）。

## 4. データフロー

- `articles/INDEX.md` を解析 → **記事一覧メタ＋遷移チャートのデータ**（日付・サイクル・支配ナラティブ・強度・主要データ・ファイルリンク）。INDEXを構造化データの単一ソースとする。
- `articles/YYYY/MM/*.md` → **個別記事ページ**としてレンダリング（計器盤テーブル・冒頭注記・末尾免責もそのまま整形）。
- 記事のタイトルは本文先頭の `# 見出し`、配信日時・強度は本文の該当行から補完（主はINDEX）。

## 5. ページ構成（v1）

| ルート | 内容 |
|---|---|
| `/` トップ | 最新6h記事の要点＋**ナラティブ遷移チャート**＋最近の記事リスト |
| `/articles/[...slug]` 記事 | 個別6h記事の整形表示（Markdown→HTML、表・引用・免責対応） |
| `/archive` アーカイブ | 全記事を日付降順で一覧（6hストリーム） |

**ナラティブ遷移チャート**：INDEXの「強度(X/10)」を時系列折れ線で表示。各点に日付・支配ナラティブをツールチップ。プロジェクト独自の“ナラティブ追跡”ビジュアル。

## 6. コンポーネント分割（単一責務）

- `lib/articles`：`articles/` 走査・Markdown読込・frontmatter無し記事のメタ抽出（タイトル/日時/強度）
- `lib/index-parser`：`INDEX.md` のテーブル→構造化データ（一覧・チャート共用）
- `lib/markdown`：Markdown→HTML（gray-matter＋remark/rehype、表・GFM対応）
- `components/NarrativeChart`：強度時系列チャート（Recharts等の軽量ライブラリ）
- `components/ArticleCard` / `ArticleList`：一覧表示
- `app/`：各ルート（SSG）

各ユニットは入出力が明確で独立テスト可能なこと。

## 7. 技術選定

- Next.js（App Router）／SSG（必要に応じてISR）
- Markdown：gray-matter ＋ remark/remark-gfm/rehype（軽量・依存少）
- チャート：軽量ライブラリ（Recharts等）
- ホスティング：Vercel Hobby（独自ドメインは後付け）／自動デプロイ
- Node 24 LTS 既定

## 8. デザイン

- **デザインスキル＝Impeccable** を採用。流れ：`shape`（UX設計）→ `craft`（実装）→ `typeset`/`layout`/`quieter`（整え）→ `audit`/`adapt`（a11y・パフォーマンス・レスポンシブ仕上げ）。
- 方針：知的で落ち着いたエディトリアル。長文＋表が主役なのでタイポと可読性を最優先。「煽らない」プロジェクト規定に整合（過剰演出を避ける）。
- ※デザインスキルの起動は実装段階（計画のあと）。

## 9. トーン・規定（プロジェクトCLAUDE.md準拠）

断定しない／価格予想しない／売買助言しない／スコアを煽りに使わない／各記事末尾に免責表示。冒頭の「⚠️自動生成」注記もそのまま表示。

## 10. スコープ外（v1.1以降）

- 週次まとめルーチン（別ルーチンとして後追い追加。日曜にその週の6h記事を統合し自動公開）
- 収益化（Stripe等）／検索・タグ・カテゴリ／OGP最適化の作り込み／RSS

## 11. 成功基準

- ルーチンの記事pushで、サイトが自動再デプロイされ最新記事が反映される。
- トップ・記事・アーカイブが正しく表示され、遷移チャートがINDEXの強度推移を可視化する。
- 計器盤テーブル・免責・トーンが崩れず表示される。
- モバイルで読みやすい。
- 人手の運用作業は「見る・必要なら手直し」のみ（公開は自動）。
