# 週次メタナラティブ統合 — 設計

- **作成日**: 2026-06-19
- **対象**: narrative-broadcast（クリプト・ナラティブ観測メディア）
- **種別**: 新機能（新しい記事タイプ＋自動生成パイプライン）

## ゴール

週1回、その週の全銘柄6h記事を入力に、**BTC/ETH/SOL/XRP を横断したメタナラティブ統合記事を1本**自動生成・公開する。日々の銘柄別6h記事（1日16本）が出さない「全体像」——今週、市場の語りが横断的にどう動いたか——を与えるフラッグシップ記事とする。

### 設計判断（確定事項）

| 論点 | 決定 |
|---|---|
| 記事の正体 | 週のメタナラティブ統合（4銘柄**横断1本**）。中に銘柄別の週間強度ミニ表を挟む |
| 生成方式 | 完全自動（GitHub Actions × LLM、週1回）。6hパイプラインと同じ堅さ |
| 入力ソース | 今週の全銘柄6h記事を再読み込み。W/W差分は**記事内の計器盤から導出**（新規fetchゼロ） |
| 配信タイミング | 日曜 21:00 JST。「今週＝月曜00:00〜日曜の生成時点」 |
| サイト配置 | weekly を**新カテゴリ化**（`/c/weekly`）。ホームのピン留めは将来の追い足し |
| 合成強度 | 週末時点の4銘柄強度の平均（四捨五入で1〜10）。前週平均との差で前週比 |

## 非ゴール（YAGNI）

- 銘柄別の週次4本化（横断1本に集中）。
- 週末時点の最新生データの別途取得（記事内計器盤から導出で十分・壊れにくい）。
- ホームhero周りのピン留め改修（価値確認後に別途）。
- ナラティブチャートへの合成強度系列の追加（単位が異なるため当面はチャート除外）。

## アプローチ

### 1. 入力とデータ導出（新規fetchゼロ）

- **収集**: 今週（JST月曜00:00〜生成時点）に該当する `articles/YYYY/MM/*.md` を全銘柄ぶん集める（最大28本＝4銘柄×7日強）。
- **パース**: 各記事から `タイトル / ナラティブ強度 / 配信日時 / 銘柄 / 計器盤テーブルの実測値` を抽出。タイトル・強度・日時・slugは既存 `lib/index-parser.ts`／`pipeline/publish.ts` のヘルパーで取得。計器盤テーブル（本文末の `| 指標 | 実測値 | ソース |`）は新規パーサで読む。
- **W/W導出**: 銘柄ごとに「週初の記事」と「週末の記事」の計器盤を引き算し、価格W/W%・強度W/W・トレンド語の入れ替わりを算出。**外部APIを叩かない**＝失敗モードもデータ/本文のズレも発生しない。
- **合成強度**: 週末時点の各銘柄の最新強度の平均（四捨五入で1〜10）。前週の同値との差で「前週比」を出す。記事には `**ナラティブ強度：X/10**` 形式で書き、既存 `extractStrength` → INDEX の流れに無改修で乗せる。

### 2. 記事の構成

1. タイトル（今週の統合メタナラティブを1行）
2. **合成強度 X/10（前週比 ±Y）**
3. リード（2-3文：今週、市場の語りに何が起きたか）
4. 今週の支配的メタナラティブ（全4銘柄を貫いた力学）
5. **銘柄別・週間強度スナップ（ミニ表）**: BTC/ETH/SOL/XRP × 週初→週末・主ナラティブ・W/W差分
6. ナラティブの伝染・ローテーション（銘柄間で物語がどう移った/収斂したか）
7. 構造分析・週次版（表層/中層/深層の**週間の変化** ＋ reflexivity判定）
8. 今週の決定的変化点（3つ）
9. 来週の観測ポイント（語りの分岐点。価格予想・売買助言は禁止＝6hと同じ規律）
10. **週間計器盤**（W/W差分テーブル：各銘柄の価格W/W・強度W/W・トレンド語の入替）
11. 出典・免責

### 3. パイプライン実装

新規ディレクトリ `pipeline/weekly/`：

- `collect-week.ts`: 今週の記事md収集＋パース（週境界＝JST月曜00:00基準）。返り値は銘柄ごとの記事時系列＋計器盤スナップ。
- `weekly-data-table.ts`: W/W計器盤テーブルの組み立て。
- `build-weekly-prompt.ts`: 週次専用の統合プロンプト（横断・メタ視点。6hの構造分析規律を踏襲）。
- `run-weekly.ts`: 収集 → 生成 → `pipeline/publish.ts` のヘルパーで公開。`--no-publish` でドライラン（`/tmp` 出力）。

**ファイル命名**: `YYYY-MM-DD-HHMM-7d-weekly.md`（cycle=`7d`／接尾辞 `-weekly`）。`articleRelPath` 相当のロジックで `articles/YYYY/MM/` に保存。`buildIndexRow`/`insertIndexRow` 再利用でINDEX.mdにも1行載る（サイクル列＝`7d`）。

**LLM生成**: 既存 `pipeline/generate.ts` の `generateArticle` を再利用（同じAPIキー経路）。

### 4. サイト配置

- `lib/categories.ts` に **weekly カテゴリ**を1件追加（`slug: 'weekly'` / label「週次メタナラティブ」/ short「週次」/ 独自のbrand色・symbol）。
- `/c/weekly`・`sitemap.ts`・フィードのバッジは接尾辞ベースのため**自動で機能**する見込み。
- **実装時に要確認**: `lib/categories.ts` の `categoryFromSlug` が `-weekly` 接尾辞を正しく weekly カテゴリへ解決すること。既存は銘柄接尾辞前提のため、横断カテゴリで分岐が必要なら最小修正を加える。
- ホームの遷移チャート（`app/page.tsx` の `series`）は CATEGORIES を写すため、weekly を**チャート系列から除外**するフィルタを1行入れる（合成強度は銘柄別強度と単位が異なる）。フィードと `/c/weekly` には表示する。

### 5. スケジューリング

- 新規 `.github/workflows/publish-weekly.yml`（`workflow_dispatch` トリガー）。`npx tsx pipeline/weekly/run-weekly.ts` を実行し main に直接コミット&プッシュ（既存 `publish-btc.yml` のコミット手順を踏襲）。
- cron-job.org に **日曜21:00 JST** のトリガーを1本追加（既存の外部cron運用に追従）。
- `concurrency.group` を `publish-btc` と共有 → INDEX.md の同時書き込み衝突を回避。

## エラーハンドリングとエッジケース

- **銘柄の記事欠落**: ある銘柄の記事が週内に1本も無い場合、ミニ表・計器盤で欠落を注記しグレースフルに継続（その銘柄を除いて統合）。
- **初週（前週なし）**: 合成強度の前週比は `±0`。
- **記事が極端に少ない週**: 統合の素材が乏しい場合も、ある分だけで生成（破綻させない）。
- **計器盤パース失敗**: 個別指標が読めない場合は当該セルを `N/A` 扱いにし、他は継続。
- **LLM出力の強度欠落**: `extractStrength` が拾えない場合は合成強度（算出値）でフォールバックしてINDEX行を作る。

## テスト

- 週境界の選別（JST月〜日の範囲に入る記事だけを集めること）。
- 計器盤テーブルのパース（表記揺れ・欠損セル）。
- W/W導出（週初/週末スナップの引き算）。
- 合成強度の計算（平均・四捨五入・前週比）。
- weekly INDEX行の生成（`buildIndexRow` 互換）。
- ドライラン（`--no-publish`）が `/tmp` に出力し main を汚さないこと。
